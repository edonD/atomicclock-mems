Overview

What this project is

A distributed sensor network that fuses passive MEMS sensors to answer one question: what is moving through this zone, where, and with what confidence?

System concept

Put six to eight low-power nodes around a fence line, trail, utility yard, or remote work site. Each node listens only for coarse signatures: sound envelope, ground vibration, pressure changes, optional local tamper motion, and optional thermal blobs. The node does edge processing locally and sends only features or alerts to a gateway.

The gateway correlates time-aligned packets from multiple nodes, estimates the event location, classifies the event at a coarse level, and renders a zone map. In a practical prototype, the useful labels are usually:

  • person: periodic acoustic and footfall pattern, moderate vibration, weak magnetic disturbance
  • vehicle: strong broadband acoustic energy, strong vibration, clear magnetic disturbance
  • animal: softer, irregular signature with weaker vibration and low magnetic response
  • wind or nuisance: acoustic change without the expected cross-modal pattern
  • tamper: strong local IMU spike on a single node rather than a moving track
Build boundary: the recommended design is for authorized property, infrastructure, and wildlife monitoring. Store features, not raw conversations. Keep the output coarse and non-identifying unless you have a real legal and ethical basis to go further.

Operating loop

1
Sense locally
MEMS mic, accelerometer, pressure sensor, optional gyro and thermal pixel array.
2
Extract features
RMS, band energy, step periodicity, vibration burst score, pressure transient, thermal occupancy.
3
Filter nuisance events
Reject wind, rain, fence rattle, or isolated node hits when the multi-sensor pattern does not line up.
4
Fuse across nodes
Gateway correlates timestamps and weighted node confidence to estimate track and class.
5
Render a coarse site picture
Zone alert, confidence, track, node health, and last event timeline.

    Build

    How to build a practical prototype

    Start with one node plus one gateway, prove local feature extraction, then expand to a six-node network.

    Node hardware

    • MCU: ESP32-S3, STM32, or nRF52 class part
    • Digital MEMS microphone on I2S or PDM
    • Low-noise MEMS accelerometer on SPI or I2C
    • MEMS barometer for wind and pressure-transient context
    • Optional gyro for tamper or mounting vibration
    • Optional thermal microbolometer module for non-identifying heat blobs
    • Radio: LoRa for long range or Wi-Fi for short range demo
    • Battery + solar or fixed 12 V supply

    Gateway

    • Raspberry Pi or small x86 box
    • LoRa concentrator hat or standard Wi-Fi access point
    • Time alignment and event fusion service
    • Simple dashboard server with map, node health, and timeline
    • Database for feature packets and alerts only

    Software split

    • Edge: denoise, windowing, thresholds, short feature vectors
    • Fusion: cross-node time correlation and weighted centroid estimate
    • UI: recent alerts, heat zones, uptime, battery, and nuisance rate

    Recommended build sequence

    1. Build one node on the bench. Bring up mic, accelerometer, and barometer. Log data locally first.
    2. Create edge features. Compute 250 ms windows of audio RMS, band energy, vibration RMS, pressure delta, and IMU tamper flags.
    3. Do not stream raw audio by default. Send only feature vectors and alerts.
    4. Label a small dataset. Walk a person past the node, roll a cart or vehicle nearby, generate wind or fan noise, and record nuisance cases.
    5. Add a second node. Verify that the same event appears on both nodes with a time offset and different amplitude.
    6. Add a gateway. Fuse packets inside a 1 to 3 second window and compute a coarse location estimate.
    7. Expand to six nodes. Place them around the site perimeter or a path corridor.
    8. Add the dashboard. Show zone map, event type, confidence, node battery, last communication, and nuisance rejection stats.
    9. Only then consider thermal. A thermal MEMS module helps with confirmation but raises cost and integration effort.

    Simple edge firmware sketch

    loop every 250 ms:
  mic_window = read_audio_samples()
  imu_window = read_accel_samples()
  pressure = read_barometer()

  features = {
    audio_rms,
    audio_band_low,
    audio_band_mid,
    step_periodicity,
    vibration_rms,
    vibration_burst_score,
    pressure_delta,
    tamper_flag
  }

  local_score = rules_or_small_model(features)

  if local_score > threshold:
    send_packet({
      node_id,
      timestamp,
      features,
      local_score,
      battery,
      temperature
    })

    Simple gateway fusion sketch

    collect packets over 1 to 3 second window
group by overlapping time and compatible class

for each event cluster:
  estimate location = weighted centroid of node scores
  estimate class = multi-sensor rule set or light model
  reject if:
    only one noisy node triggered
    pressure says strong gust and vibration is absent
    acoustic energy is high but cross-node timing is inconsistent

publish:
  class, confidence, location, triggered_nodes, node_health
    MEMS

    Which MEMS are used and why

    The best first prototype uses only a few MEMS parts. Add the expensive ones later.

    Digital MEMS microphone

    This is the most important sensor for the first prototype. It gives you activity detection, spectral shape, cadence, and nuisance classification.

    Use it for: footsteps, wheels, motors, voices as anonymous activity bands, fence rattle.

    MEMS accelerometer

    This turns your node or mounting stake into a compact vibration sensor. It is what separates real movement from pure airborne noise.

    Use it for: footfall bursts, vehicle rumble, tamper detection, pole vibration.

    MEMS pressure sensor

    It does not detect people directly very well, but it gives environmental context that helps you reject gusts, enclosure leaks, and weather-driven false alarms.

    Use it for: wind filtering, enclosure health, context features.

    MEMS thermal microbolometer

    This is optional. It raises cost sharply, but it gives a privacy-preserving thermal occupancy channel. You can confirm presence without seeing faces or text.

    Use it for: thermal confirmation, night operation, coarse blob tracking.

    Important precision note

    Not every useful sensor in a practical system is MEMS. Radios, MCUs, storage, and many magnetometers are microelectronic but not always MEMS in the strict mechanical sense. If you want a clean "mostly MEMS" build, use this stack:

    • Core MEMS stack: digital MEMS microphone + MEMS accelerometer + MEMS pressure sensor
    • Optional MEMS stack: MEMS gyro + MEMS thermal microbolometer
    • Non-MEMS support parts: MCU, LoRa radio, power management, gateway computer
    Limits

    Where this concept is good and where it is weak

    Good fits

    • Authorized perimeter awareness
    • Utility yards and remote infrastructure
    • Wildlife corridor monitoring
    • Temporary construction or research sites
    • Nuisance filtering before a human reviews the alert

    Weak spots

    • High wind, heavy rain, and loose mounting points
    • Dense urban noise with strong reverberation
    • Trying to identify individuals instead of coarse classes
    • Single-node deployments with no fusion
    • Expecting thermal without paying the power and cost penalty

    Recommended boundaries

    • Feature-only telemetry by default
    • No hidden recording of conversations
    • Post signage on monitored property
    • Keep classifications coarse and auditable
    • Use a review step before any consequential action
